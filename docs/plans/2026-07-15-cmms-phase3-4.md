**Conflict detection rule:**
```typescript
async assign(orderId: string, roomId: string) {
  const order = await repo.findById(orderId);
  // Check date overlap with other orders on same room
  const conflicts = await repo.findRoomBookingsInRange(
    roomId, order.checkInAt, order.checkOutAt
  );
  const realConflicts = conflicts.filter(c => c.id !== orderId);
  if (realConflicts.length > 0) {
    throw new ConflictError(
      `房间 ${roomId} 在 ${order.checkInAt.toISOString()} ~ ${order.checkOutAt.toISOString()} 已被预订`
    );
  }
  // Validate room capacity vs occupants (main + subordinates + roommates)
  const totalOccupants = 1 + order.roommateIds.length;
  const room = await hotelRepo.findRoomById(roomId);
  if (room.roomType === 'SINGLE' && totalOccupants > 1) {
    throw new ValidationError(`单人房不能容纳 ${totalOccupants} 人`);
  }
  return repo.assign(orderId, roomId);
}
```

**Date overlap SQL pattern:**
```typescript
// Two ranges [A_start, A_end) and [B_start, B_end) overlap if:
// A_start < B_end AND B_start < A_end
await prisma.lodgingOrder.findMany({
  where: {
    hotelRoomId: roomId,
    status: { notIn: ['CANCELED'] },
    AND: [
      { checkInAt: { lt: checkOutAt } },   // existing.start < new.end
      { checkOutAt: { gt: checkInAt } },   // existing.end > new.start
    ],
  },
});
```

```typescript
async assign(orderId: string, tableId: string) {
  const table = await repo.findTableById(tableId);
  // Count existing assignments for same mealTime + table
  const order = await repo.findById(orderId);
  const existing = await repo.countTableAssignments(tableId, order.mealTime);
  if (existing + 1 > table.capacity) {
    throw new ValidationError(
      `餐桌 ${table.name} 容量 ${table.capacity}，已分配 ${existing} 人`
    );
  }
  return repo.assign(orderId, tableId);
}
```

```typescript
async deliver(orderId: string) {
  const order = await repo.findById(orderId);
  const gift = await giftRepo.findById(order.giftId);
  if (gift.stock < order.quantity) {
    throw new ConflictError(`礼品「${gift.name}」库存不足（剩 ${gift.stock}，需 ${order.quantity}）`);
  }
  // Transaction: decrement stock + mark delivered
  await prisma.$transaction([
    prisma.gift.update({
      where: { id: gift.id },
      data: { stock: { decrement: order.quantity } },
    }),
    prisma.giftOrder.update({
      where: { id: orderId },
      data: { status: 'DELIVERED', deliveredAt: new Date() },
    }),
  ]);
}
```

**Aggregator:**
```typescript
export interface Guest360View {
  meetingGuest: MeetingGuest & { guest: Guest; primary?: MeetingGuest; subordinates: MeetingGuest[] };
  transport: TransportOrder[];
  lodging: LodgingOrder[];
  catering: CateringOrder[];
  gift: GiftOrder[];
  companion: CompanionAssignment[];
  fees: FeeRecord[];
  agenda: AgendaItem[];  // where this guest is speaker
  stats: {
    totalTasks: number;
    completedTasks: number;
    pendingTasks: number;
    totalFeeAmount: number;
  };
}

async function getGuest360(meetingGuestId: string): Promise<Guest360View> {
  const [
    meetingGuest, transport, lodging, catering, gift, companion, fees, agenda
  ] = await Promise.all([
    meetingGuestRepo.findWithRelations(meetingGuestId),
    transportRepo.listByMeetingGuest(meetingGuestId),
    lodgingRepo.listByMeetingGuest(meetingGuestId),
    cateringRepo.listByMeetingGuest(meetingGuestId),
    giftRepo.listByMeetingGuest(meetingGuestId),
    companionRepo.listByMeetingGuest(meetingGuestId),
    feeRepo.listByMeetingGuest(meetingGuestId),
    agendaRepo.listBySpeaker(meetingGuestId),
  ]);

  const allTasks = [...transport, ...lodging, ...catering, ...gift];
  return {
    meetingGuest, transport, lodging, catering, gift, companion, fees, agenda,
    stats: {
      totalTasks: allTasks.length,
      completedTasks: allTasks.filter(t => isCompleted(t)).length,
      pendingTasks: allTasks.filter(t => isPending(t)).length,
      totalFeeAmount: fees.reduce((sum, f) => sum + Number(f.amount), 0),
    },
  };
}
```

```typescript
export async function rateLimit(
  identifier: string,
  limit: number,
  windowSec: number,
): Promise<{ allowed: boolean; remaining: number }> {
  const key = `ratelimit:${identifier}`;
  const now = Date.now();
  const windowStart = now - windowSec * 1000;
  
  const redis = getRedis();
  const pipeline = redis.pipeline();
  pipeline.zremrangebyscore(key, 0, windowStart);
  pipeline.zadd(key, now, `${now}:${Math.random()}`);
  pipeline.zcard(key);
  pipeline.expire(key, windowSec);
  const results = await pipeline.exec();
  
  const count = results![2][1] as number;
  return { allowed: count <= limit, remaining: Math.max(0, limit - count) };
}
```

## Appendix: Phase 3+4 Architecture Decisions

### A.1: Why double-entry for resources (Resource + Order tables)

Each reception type has 2 tables:
- Resource: `Vehicle`, `HotelRoom`, `DiningTable`, `Gift`, `Companion` (the thing being assigned)
- Order: `TransportOrder`, `LodgingOrder`, `CateringOrder`, `GiftOrder`, `CompanionAssignment` (the assignment record)

Rationale:
- Resource outlives any single meeting (vehicles, hotels persist)
- Order is per-meeting per-guest
- Conflict detection queries Orders for a given Resource
- Cleaner than polymorphic single table

### A.2: Why FeeRecord is auto-generated from hooks

Cross-module fees are error-prone if manually entered. By hooking into:
- TransportOrder.markCompleted
- LodgingOrder.checkOut
- CateringOrder.finish
- GiftOrder.deliver

We ensure no fee leaks. Manual FeeRecord still allowed for ad-hoc expenses.

### A.3: Why notification uses template-in-DB (not hardcoded)

- Operations team can edit templates without deploy
- A/B test different wordings
- Multi-tenant future-proofing (different orgs different templates)
- Audit trail (template changes are logged)

### A.4: Why audit log uses Prisma extension (not service-layer)

Service layer would require every method to manually call `logAction()`. Easy to forget. Prisma `$extends` interceptor catches all writes automatically.

Trade-off: harder to log business-level actions (like "login" which isn't a DB write). For those, manual `logAction()` call in the action handler.

### A.5: Why no WebSocket for real-time updates

Per project rule (`CLAUDE.md` prohibits WebSocket). TanStack Query polling is simpler:
- Stateless backend (easier to scale)
- No connection management
- Tolerates network blips better
- Trade-off: up to 5-30s latency (acceptable for reception operations)

### A.6: Why non-root container + read-only FS

Defense in depth:
- Even if attacker gets RCE, can't modify app code
- Can't write to /etc or install packages
- Memory limit prevents resource exhaustion attacks
- `no-new-privileges:true` blocks privilege escalation

Standard hardening, no real downside.
