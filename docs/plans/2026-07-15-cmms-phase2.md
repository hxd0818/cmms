# CMMS Phase 2 Implementation Plan (Meeting + Reception Core)

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.
>
> **Established Patterns:** Phase 1 (Guest module) is the canonical reference. Read `docs/plans/2026-07-15-cmms-foundation.md` Task 1.1-1.14 to understand the layered pattern (Prisma → Zod → Repo → Service → Actions → UI → tests).

**Goal:** 实现 Phase 2 的现场运营核心：Meeting（会议）、MeetingGuest（会议-嘉宾关联 + 随行）、Agenda（议程）、Reception（签到）、Transport（接送调度）、Driver 端（司机扫码）。完成后系统可完整支持"一场会议"从筹备到现场接待的全流程。

**Architecture:** 沿用 Phase 1 的分层架构。MeetingGuest 是接待任务枢纽（所有 Transport/Lodging/Catering/Gift 通过 meetingGuestId 关联），并维护随行人员主从关系（规格继承）。引入有状态 Token（HMAC + Redis + DB）支持嘉宾/陪同端 + 司机端扫码访问。

**Tech Stack:** Next.js 16 + Prisma 7 + PostgreSQL + Redis 7 + BullMQ + Zod 4 + CASL + NextAuth v5 + shadcn/ui 4.12

**Reference Design:** `docs/plans/2026-07-15-cmms-design.md` Sections 2 (Domain Model), 3 (RBAC + Workflows + State Machines)

export type MeetingStatus = z.infer<typeof meetingStatusSchema>;

export const rsvpStatusSchema = z.enum(['PENDING', 'CONFIRMED', 'DECLINED']);
export const receptionStageSchema = z.enum([
  'NOT_ARRIVED', 'CHECKED_IN', 'IN_HOUSE', 'DEPARTED', 'NO_SHOW',
]);
export const entourageRoleSchema = z.enum([
  'PRIMARY', 'SECRETARY', 'SECURITY', 'INTERPRETER', 'FAMILY', 'AIDE', 'DRIVER',
]);
export type EntourageRole = z.infer<typeof entourageRoleSchema>;

export const meetingCreateSchema = z.object({
  name: z.string().min(1).max(200),
  code: z.string().min(1).max(50).regex(/^[A-Z0-9-]+$/i, '编号仅允许字母数字和连字符'),
  startAt: z.coerce.date(),
  endAt: z.coerce.date(),
  venue: z.string().max(200).optional(),
  description: z.string().max(2000).optional(),
}).refine((d) => d.endAt > d.startAt, { message: '结束时间必须晚于开始时间', path: ['endAt'] });

export type MeetingCreateInput = z.infer<typeof meetingCreateSchema>;
export const meetingUpdateSchema = meetingCreateSchema.partial();
export type MeetingUpdateInput = z.infer<typeof meetingUpdateSchema>;

export const meetingGuestCreateSchema = z.object({
  meetingId: z.string().cuid(),
  guestId: z.string().cuid(),
  groupTags: z.array(z.string()).default([]),
  primaryMeetingGuestId: z.string().cuid().optional(),
  entourageRole: entourageRoleSchema.optional(),
  levelOverride: guestLevelSchema.optional(),
  inheritLodging: z.boolean().default(true),
  inheritTransport: z.boolean().default(true),
});

async create(data: AgendaCreateData) {
  // For each speaker in speakerIds, check no overlapping agenda item
  for (const speakerId of data.speakerIds) {
    const conflicts = await repo.findSpeakerItemsInRange(
      speakerId, data.startAt, data.endAt

**Conflict detection rule:**
```typescript
async assign(orderId: string, vehicleId: string) {
  const order = await repo.findById(orderId);
  // Time window: ±30 minutes around pickupTime
  const windowStart = new Date(order.pickupTime.getTime() - 30 * 60 * 1000);
  const windowEnd = new Date(order.pickupTime.getTime() + 60 * 60 * 1000);
  const conflicts = await repo.findVehicleBookingsInRange(vehicleId, windowStart, windowEnd);
  if (conflicts.length > 0) {
    throw new ConflictError(`车辆 ${vehicleId} 在此时间段已被占用`);
  }
  // Capacity check: if main guest has inheritTransport subordinates, total seats needed = 1 + subordinates
  const subordinates = await meetingGuestRepo.findSubordinates(order.meetingGuestId);
  const totalNeeded = subordinates.length + 1;
  const vehicle = await vehicleRepo.findById(vehicleId);
  if (vehicle.capacity < totalNeeded) {
    throw new ValidationError(`车辆容量 ${vehicle.capacity} < 需求 ${totalNeeded}（含随行）`);
  }
  return repo.assign(orderId, vehicleId);
}
```

// lib/auth/tokens.ts
import crypto from 'crypto';

const TOKEN_SECRET = process.env.NEXTAUTH_SECRET!; // reuse

export async function generateToken(): Promise<{ raw: string; hash: string }> {
  const raw = crypto.randomBytes(32).toString('base64url');
  const hash = hashToken(raw);
  return { raw, hash };
}

export function hashToken(raw: string): string {
  return crypto.createHmac('sha256', TOKEN_SECRET).update(raw).digest('hex');
}

export async function verifyGuestToken(
  raw: string,
): Promise<{ meetingGuestId: string } | null> {
  const hash = hashToken(raw);
  const record = await prisma.guestAccessToken.findUnique({
    where: { tokenHash: hash },
  });
  if (!record) return null;
  if (record.revokedAt) return null;
  if (record.expiresAt < new Date()) return null;
  // Async update last access
  void prisma.guestAccessToken.update({
    where: { id: record.id },
    data: { lastAccessedAt: new Date(), accessCount: { increment: 1 } },
  });
  return { meetingGuestId: record.meetingGuestId };
}

## Appendix: Phase 2 Architecture Decisions

### A.1: Why MeetingGuest as the hub (not Guest + meetingId)?

All transport/lodging/catering/gift orders reference `meetingGuestId` (not `guestId + meetingId`). Rationale:
- Single index (vs composite)
- Enforces "guest must be registered for the meeting before assigning tasks"
- ReceptionStage lives on MeetingGuest (per-meeting state, not per-guest)
- Cascade delete: removing meeting cleans up all related orders

### A.2: Entourage relation as self-reference on MeetingGuest

`primaryMeetingGuestId` (nullable FK to MeetingGuest) supports:
- Same person as PRIMARY in meeting A, SECRETARY in meeting B (different rows)
- `levelOverride` per meeting (normally inherits from Guest)
- `inheritTransport` flag lets dispatch system auto-group main + subordinates

### A.3: Token is stateful (DB-backed), not JWT

| Approach | Pros | Cons |
|---|---|---|
| Stateless JWT | No DB lookup per request | Cannot revoke (guest phone lost = leak forever) |
| Stateful DB token | One-click revoke, audit trail | 1 DB query per portal request |

For sensitive VIP data, revocability wins. AccessCount + LastAccessedAt give us anomaly detection for free.

### A.4: Driver portal is separate from guest portal

Drivers are typically NOT in the Guest table (external fleet). Their token binds to `TransportOrder`, not `MeetingGuest`. Two portals, two token types, one shared HMAC mechanism.

### A.5: Polling instead of WebSocket for live updates

Per CLAUDE.md project rule: "严禁在项目中使用 websocket". Use TanStack Query with 5s `refetchInterval` on Kanban/reception pages. Trade-off: slight latency (max 5s) for simplicity and stateless backend.
